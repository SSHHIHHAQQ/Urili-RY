# 商城商品审核流程接入计划

## 1. 现有审核表结论

当前商品审核已经有一组可复用的审核表，第一版不建议另起审核事实表。

- `product_review_request`：审核主单，记录审核类型、状态、SPU、卖家、提交人、审核人、价格/仓库/差异摘要、待审冲突键等。
- `product_review_item`：审核明细，按 SPU/SKU 拆分变更项，记录新增、修改、调价等变更对象。
- `product_review_snapshot`：审核快照，保存 `BEFORE`、`AFTER`、`DIFF` 的 JSON 载荷和 hash。
- `product_review_operation_log`：审核操作日志，记录提交、通过、驳回、撤回等动作。

现有表结构已经支持“审核前快照 + 审核后快照”的模式，适合承载商品编辑审核。当前主要缺口不是表，而是服务逻辑：现在只实现了草稿商品提交审核后通过变为待上架，以及部分新增 SKU 审核逻辑；已审核商品的编辑还没有走审核快照和审批生效。

## 2. 推荐核心原则

审核后的商品不能把编辑内容先写入正式商品事实表，否则审核驳回时要么丢用户填写内容，要么污染买家正在看到的正式商品。

推荐规则：

- 草稿状态：任意编辑直接保存到商品正式表，不需要审核。
- 草稿提交审核：创建 `NEW_PRODUCT` 审核单，商品仍保持草稿；审核通过后改为待上架，驳回后仍是草稿，用户已填写内容不丢。
- 待上架、已上架、已下架商品编辑：提交后只写审核快照，不立即覆盖正式商品表。
- 审核通过：把 `AFTER` 快照一次性应用到正式商品表。
- 审核驳回：正式商品保持审核前状态，用户提交的新内容保留在审核快照里，可用于后续“基于驳回稿继续编辑/重新提交”。
- 同一个 SPU 存在待审单时，继续使用 `active_pending_key` 锁定，避免多份待审稿互相覆盖。

## 3. 状态流转建议

### 草稿商品

1. 新增或编辑草稿：直接保存。
2. 提交审核：生成 `NEW_PRODUCT` 审核单。
3. 审核通过：SPU/SKU 状态从草稿变为待上架。
4. 审核驳回：SPU/SKU 仍为草稿，用户可继续编辑后重新提交。

### 待上架商品

1. 编辑商品资料、类目属性、图片、详情图文、SKU、仓库、来源绑定等：生成编辑审核单。
2. 审核期间列表仍展示旧版本正式数据，并显示存在待审变更。
3. 审核通过后覆盖正式数据，状态仍为待上架。
4. 审核驳回后正式数据不变，驳回稿保留。

### 已上架商品

1. 编辑后不影响买家端当前看到的商品。
2. 审核通过后新版本立即生效到正式商品，状态仍为已上架。
3. 审核驳回后买家端始终看到旧版本，卖家填写内容保留在驳回审核单。

### 已下架商品

1. 编辑同样需要审核。
2. 审核通过后商品信息更新，但状态仍为已下架。
3. 是否重新上架仍走上架流转，不和编辑审核混在一起。

### 停用商品

停用是独立管控状态。停用期间建议不允许提交商品编辑审核，只允许管理端恢复或处理管控原因。恢复后再按原销售状态继续审核/编辑。

## 4. 审核类型拆分

现有字典已有：

- `NEW_PRODUCT`：新商品审核。
- `ADD_SKU`：新增 SKU 审核。
- `EDIT_PRODUCT_INFO`：SPU 信息编辑审核。
- `EDIT_SKU_INFO`：SKU 信息编辑审核。
- `EDIT_PRICE`：调价审核。

第一版建议按提交内容自动归类：

- 只改销售价：`EDIT_PRICE`。
- 只新增 SKU：`ADD_SKU`。
- 只改 SKU 规格、图片、尺寸重量、供货价、仓库、来源绑定：`EDIT_SKU_INFO`。
- 改 SPU 标题、类目、属性、商品图、详情图文：`EDIT_PRODUCT_INFO`。
- 同时涉及多类变更：主单使用 `EDIT_PRODUCT_INFO`，明细里按 SPU/SKU 和 payload 类型展示差异。

删除 SKU 暂不建议混在普通编辑里静默处理。第一版可以先禁止已审核商品删除 SKU，后续如果要支持，应补充 `DELETE_SKU` 审核类型或把删除作为 `EDIT_SKU_INFO` 下的明确删除明细。

## 5. 后端实施计划

1. 扩展商品保存入口。
   - 草稿商品继续直接保存。
   - 非草稿商品保存时，不写正式表，改为创建编辑审核单。
   - 编辑页保存成功后提示“已提交审核”，返回列表或停留当前页按后续确认。

2. 建立完整快照构建能力。
   - `BEFORE`：从当前正式商品表读取完整 SPU/SKU/属性/图片/详情/仓库/来源绑定/价格。
   - `AFTER`：从前端提交 payload 规范化后生成。
   - `DIFF`：生成审核页可读差异摘要。

3. 审核通过应用快照。
   - 审批时重新计算当前正式数据 hash，确认仍等于 `BEFORE` hash。
   - 若一致，则事务内应用 `AFTER` 到正式商品表。
   - 若不一致，审批失败并提示数据已变化，避免覆盖其他变更。

4. 审核驳回保留稿件。
   - 只更新审核单状态和日志。
   - 不改正式商品表。
   - 保留 `AFTER` 快照，后续前端可读取驳回稿继续编辑。

5. 操作日志。
   - 商品操作日志记录“提交编辑审核”“审核通过生效”“审核驳回”。
   - 审核操作日志继续记录审核动作。

## 6. 前端实施计划

1. 商品编辑页。
   - 草稿编辑仍显示“保存草稿/提交审核”。
   - 非草稿编辑显示“提交审核”，不再表现成普通保存。
   - 若存在待审单，禁止重复提交，并给出查看审核单入口。

2. 商品列表。
   - 增加“待审变更”标识或字段。
   - 已上架/待上架/已下架编辑后，列表仍展示正式版本，不展示未通过的新版本。

3. 商品审核页。
   - 复用现有商品审核菜单。
   - 强化审核详情：按 SPU 基础信息、SKU、图片、详情图文、类目属性、仓库、来源绑定、价格分组展示差异。
   - 审核通过/驳回继续使用现有按钮权限和操作日志。

4. 驳回稿处理。
   - 第一版建议先在审核详情里展示驳回稿。
   - 后续再做“基于驳回稿继续编辑/重新提交”，避免用户重新填写。

## 7. 需要确认的问题

1. 非草稿商品编辑后，按钮文案是否统一叫“提交审核”，而不是“保存”。
2. 驳回稿第一版是否只保留并展示，还是立刻支持“一键带入编辑页继续修改”。
3. 已审核商品删除 SKU 是否第一版先禁止，等后续单独设计删除 SKU 审核。
4. 商品已上架时，审核通过后的编辑内容是否立即对买家端生效；当前建议是立即生效。
