# 物流商管理需求理解与流程设计草案

日期：2026-06-10

## 当前结论

物流商管理菜单的核心不是价格策略或履约决策，而是统一管理平台与外部物流 API 的接入、授权、渠道候选、面单生成、面单取消、面单获取/下载和请求日志。

这里的“物流商”包括两类：

- 直连最终承运商，例如 UPS、FedEx、USPS 等。
- 聚合商、中间商、折扣账号商、华人物流系统等。它们本身也叫物流商，因为平台实际调用的是它们的系统。

最终实际派送方可能仍是 USPS、UPS、FedEx、GoFo、燕文或其他尾端服务商。物流商返回的渠道名称不一定标准，例如 `USPS超快`、`美国邮政`、`U-GA` 可能都需要人工映射为 USPS 体系下的某类系统渠道。

## 范围边界

第一版物流商管理只做以下能力：

- 物流商 API 接入配置。
- 最少变动字段的授权信息录入、校验、加密保存和脱敏展示。
- 从物流商系统拉取原始渠道列表，保存为物流商渠道。
- 由运营把物流商渠道映射成系统渠道，并维护最终承运商归属。
- 对外提供获取费用能力，但不在物流商管理里做比价、最优选择、费用看板或结算记录。
- 对外提供指定渠道生成面单能力。
- 保存面单生成结果，尽量保留外部系统后续调用需要的信息。
- 支持取消面单。
- 支持获取或下载已有面单。
- 保存追加型、脱敏的外部请求日志。

第一版不做以下能力：

- 不在物流商管理内做“最便宜/最快”的自动决策。
- 不做销售价、客户报价、加价规则、报价方案或财务结算。
- 不做轨迹查询和签收状态同步。
- 不按美国本土专门建模；物流商可以是任意国家、本土尾端、卡车或其他派送服务。
- 不由物流商管理判断发件地址、收件地址或仓库地址；调用方在下单时传入。
- 不支持卖家自带物流账号；第一版只支持平台统一物流账号。

## 已确认补充

- 面单文件格式第一版需要同时支持 PDF、ZPL、PNG。
- 获取费用结果不单独做业务查询记录，只进入脱敏请求日志。
- 创建面单时，外部调用方会传入全局唯一业务单号；物流商管理应基于该业务单号做幂等，避免网络重试导致重复下单。
- 系统渠道按全局唯一口径设计。
- 标准最终承运商需要维护为若依字典，物流商渠道映射时由运营选择标准承运商。
- 标准承运商第一版使用用户在 2026-06-10 提供的承运商清单作为初始来源；公开调研资料不再扩展本轮 seed 范围。
- 第一家接入方为 AGG56，定位为物流商聚合/打单系统，类似上游系统管理中的领星 WMS；详见 `docs/plans/2026-06-10-agg56-carrier-integration-plan.md`。

## 关键对象

### API 接入方

API 接入方是平台真实调用的外部系统，可以是 UPS/FedEx/USPS，也可以是中间商。每个接入方需要有独立适配器，负责认证、报价、下单、取消、重打、渠道同步和错误映射。

管理端页面应体现为“新增物流商接入”，但后端要按 `providerKind` 或类似标识路由到对应适配器。

### 最终承运商

最终承运商是实际提供尾端服务的承运方，例如 USPS、UPS、FedEx、GoFo、燕文。它不一定等于 API 接入方。

物流商返回的最终承运商名称和渠道名称不稳定，所以不能完全自动标准化。系统应允许运营在物流商渠道映射时手工指定标准最终承运商。

### 物流商渠道

物流商渠道是从物流商 API 原样拉回来的渠道数据。物流商渠道应保存原始 code、原始名称、物流商接入编号、原始最终承运商字段、服务范围、可用状态和原始响应摘要。

物流商渠道只代表“外部系统说它有这个服务”，不直接等于系统可用渠道。

### 系统渠道

系统渠道是 URILI 内部稳定使用的渠道编码。运营需要把一个或多个物流商渠道映射到系统渠道，并补充标准最终承运商。

后续订单、履约、报价等外部菜单不应该直接依赖外部渠道名称，而应使用系统渠道或映射后的稳定标识。

### 面单订单

面单订单是平台调用物流商生成标签后的结果快照。它不是财务流水，也不是订单事实源，但要保留外部菜单继续使用所需的数据。

建议保存：

- 平台侧请求编号。
- 调用方业务单号。
- 物流商接入编号。
- 外部渠道 code/name。
- 系统渠道 code/name。
- 标准最终承运商。
- 物流商订单号。
- tracking number。
- label 格式，例如 PDF、ZPL、PNG。
- label 文件存储地址或文件 ID。
- 物流商返回费用和币种快照。
- 收发件地址摘要。
- 包裹重量、尺寸、件数摘要。
- 外部响应摘要和脱敏原文存档引用。
- 当前状态：已生成、已取消、取消失败等。

面单相关基础能力拆成三个独立动作：

- 创建面单：调用物流商下单并生成新的面单记录。
- 取消面单：调用物流商作废已生成的面单，并更新原面单状态。
- 获取面单：返回已有面单文件或重新从物流商下载已有 label，不创建新的物流订单。

如果业务上需要“作废旧面单后重新创建一张新面单”，调用方应按“取消面单 + 创建面单”两个动作组合完成。物流商管理不单独设计“重新打单”这个混合动作，避免把重复下载误做成重复下单。

## 授权逻辑

物流商管理页面不让运营填写一大堆固定接口字段。每一种物流商适配器内置自己的模板，前端根据物流商类型展示最少变动字段。

推荐流程：

1. 运营选择物流商类型，例如某个中间商系统。
2. 系统显示该类型必需的最少字段，例如账号、key、secret、token、客户号或环境。
3. 运营提交后，后端先调用适配器做授权校验。
4. 授权校验成功后，后端加密保存凭据，只保存脱敏值用于展示。
5. 授权校验失败时，不覆盖已有可用凭据。
6. 后续重新授权走独立入口，仍然先校验再覆盖。
7. 每次授权校验都写外部请求日志，但日志必须脱敏。

这条逻辑应复用当前上游系统管理的安全原则：凭据只在后端加密落库，不在前端回显原文，请求日志追加保存，敏感字段脱敏。

## 渠道同步与映射逻辑

物流商授权成功后，可以手动触发“同步渠道”。

同步渠道只保存候选数据，不直接生成系统渠道：

1. 适配器调用物流商渠道列表接口。
2. 后端把外部渠道原始 code/name 和相关能力写入物流商渠道。
3. 已不存在的物流商渠道标记为失效，不物理删除。
4. 运营在页面中选择物流商渠道，映射到系统渠道。
5. 运营手工维护标准最终承运商，例如把 `美国邮政`、`U-GA` 归到 USPS。

这样可以避免被物流商不稳定的命名污染内部履约逻辑。

## 对外调用逻辑

物流商管理对其他菜单或模块提供能力，不自己做最终业务判断。

### 获取费用

外部系统传入发件地址、收件地址、包裹信息、可选系统渠道或物流商渠道范围。物流商管理调用一个或多个物流商适配器，返回每个物流商/渠道的报价结果。

物流商管理只返回结果，不判断哪个更便宜、哪个更快。选择逻辑由外部调用方处理。

### 生成面单

外部系统确定要用哪个物流商和哪个渠道后，调用生成面单接口。

物流商管理负责：

- 校验物流商接入有效。
- 校验渠道映射仍可用。
- 调用适配器生成面单。
- 保存面单订单快照。
- 保存 label 文件或文件引用。
- 返回 tracking number、物流商订单号、label 地址、费用快照和状态。

### 取消面单

外部系统传入平台面单编号或物流商订单号，物流商管理调用对应适配器取消面单，并更新面单订单状态。

取消失败时应保存失败原因和外部错误码，不删除原面单记录。

### 获取面单

外部系统传入平台面单编号或物流商订单号，物流商管理返回已有 label 文件。如果本地文件缺失但物流商支持重新下载，则调用对应适配器重新获取 label，并更新面单记录里的文件引用。

获取面单不得创建新的物流订单，也不得改变原面单状态。

## 后端模块建议

建议新增独立 `logistics` 后端模块，而不是放入现有 `integration` 模块。

原因：

- `integration` 当前偏上游 WMS、SKU、仓库、库存候选同步。
- 物流商管理会产生面单订单、取消面单、获取面单等履约前置能力。
- 后续订单履约模块会调用物流能力，不能反向把履约事实塞进上游系统模块。

接口可以按管理端和内部能力分开：

- 管理端：`/logistics/admin/carriers/**`，用于配置、授权、同步渠道、映射、日志查看。
- 业务调用：后续通过 service/facade 或内部 API 提供报价、生成面单、取消面单、获取面单。

## 前端页面建议

物流商管理页面第一版建议采用类似上游系统管理的结构：

- 左侧：物流商接入列表，展示名称、类型、状态、授权状态、最近授权时间、最近同步时间。
- 顶部操作：新增接入、重新授权、授权校验、启停、同步渠道。
- 详情区 tab：
  - 基础信息。
  - 物流商渠道。
  - 系统渠道映射。
  - 面单记录。
  - 请求日志。

页面上不展示费用看板，不做比价配置，不做客户价格规则。

## 待确认问题

1. 获取费用的请求是否需要保存成可查询记录，还是只写脱敏请求日志即可？
2. 面单文件优先保存 PDF、ZPL，还是两者都要支持？
3. 系统渠道是全局渠道，还是未来要按国家、仓库、物流商或客户可用范围拆分？
4. 标准最终承运商是否需要维护成独立字典，例如 USPS、UPS、FedEx、GoFo、Yanwen？

已确认：

- 获取费用只写脱敏请求日志。
- PDF、ZPL、PNG 都需要支持。
- 系统渠道第一版按全局唯一。
- 标准最终承运商需要独立字典。
