# 库存调整审核真实测试计划

日期：2026-06-09

## 1. 测试目标

本轮测试目标不是只验证代码或页面字段，而是用真实数据跑通库存调整审核从配置、库存调整、审核单生成、人工处理、自动到期到恢复现场的完整链路。

需要验证：

- 库存总览调整平台总库存时，审核策略是否真实生效。
- 审核配置里的策略组字段是否真实影响判定。
- 卖家绑定是否真实影响命中策略，且卖家绑定优先于全局绑定。
- 审核单生成后，库存是否按预期保持不立即扣减。
- 立即生效、调整生效时间、驳回、到期自动生效是否正确。
- 生效时是否按“申请退回数量”重新计算实际可退数量。
- 测试结束后，策略、绑定、审核单和库存现场可追溯，必要时可恢复。

## 2. 测试边界和授权

用户已授权：

- 可以在库存总览真实调整平台总库存。
- 可以真实创建和调整库存调整审核策略。

本计划不会直接改 WMS 或外部系统库存。

本计划优先选择测试商品和测试卖家；如果没有合适测试数据，会选择一个有来源库存和平台库存的 SKU，以最小影响方式执行。所有测试操作都写入记录。

## 3. 执行前准备

### 3.1 数据源确认

执行前必须确认：

- 后端激活 profile。
- MySQL 目标库。
- Redis 目标。
- 当前后端是否为最新 jar。
- 当前前端是否能访问真实后端。

记录内容：

- 数据源配置来源。
- 目标环境。
- 连接方式。
- 是否影响远端数据。

### 3.2 选择测试库存行

选择标准：

- `inventory_sku_warehouse_stock` 中存在平台库存或来源库存的 SKU。
- 优先选择平台总库存和来源可用库存都较高的记录，例如来源库存或平台库存接近或大于 `1000`。
- 平台锁定库存尽量为 `0`，避免影响实际可退数量判断。
- 优先选择测试卖家或低风险卖家。

记录测试基线：

- `stock_id`
- `seller_id`
- `sku_id`
- `system_sku_code`
- `warehouse_kind`
- `warehouse_name`
- `source_available_qty`
- `platform_total_qty`
- `platform_available_qty`
- `platform_reserved_qty`
- `version`

### 3.3 建立测试策略命名约定

所有测试策略统一使用前缀：

```text
TEST-库存调整审核-
```

所有测试备注统一写入：

```text
库存调整审核真实测试 2026-06-09
```

这样测试后可以明确识别和清理测试数据。

### 3.4 建立恢复点

执行前记录：

- 测试库存行原始平台总库存。
- 测试库存行原始平台在途库存。
- 原有默认策略和全局绑定。
- 原有测试卖家的策略绑定。

恢复原则：

- 测试完成后，将测试库存行恢复到执行前平台总库存，除非恢复会触发新的审核单，则使用测试专用放开策略完成恢复。
- 测试策略和绑定不直接删除，优先停用并记录；如果确认可以清理，再单独清理。
- 测试生成的审核单作为审计事实保留，不做物理删除。

## 4. 第一轮：配置保存和权限入口测试

目标：确认配置页本身可用，策略组和卖家绑定能真实保存、查询、编辑。

测试项：

1. 打开库存调整审核页面。
2. 打开“审核配置”。
3. 新增策略：
   - 模式：`DISABLED`
   - 方向：`BOTH`
   - 保留天数：`7`
   - 冷却小时：`168`
   - 自动生效：`Y`
   - 人工生效：`Y`
4. 编辑该策略为：
   - 模式：`CONDITIONAL`
   - 方向：`DECREASE`
   - 销量窗口：`[7,30]`
5. 新增卖家绑定：
   - 绑定类型：`SELLER`
   - 绑定对象：测试卖家 ID
   - 绑定到测试策略
6. 新增或确认全局绑定：
   - 绑定类型：`GLOBAL`
   - 对象 ID 应为 `0`
7. 测试非法输入：
   - 卖家绑定不填卖家 ID，应被前端或后端拒绝。
   - 销量窗口填无效字符串，应被拒绝。
   - 负数冷却小时或负数保留天数，应被后端拒绝。

通过标准：

- 策略列表能看到新增/编辑后的值。
- 绑定列表能看到卖家绑定和全局绑定。
- 非法配置不能保存成功。
- 权限不足时不会展示或不会允许配置操作。

## 5. 第二轮：不审核策略测试

目标：验证 `DISABLED` 真正放开，不生成审核单。

配置：

- 测试卖家绑定到 `DISABLED + BOTH` 策略。

操作：

1. 将测试库存行平台总库存调整到稳定基线，例如 `1000`。
2. 从 `1000` 下调到 `150`。
3. 再从 `150` 上调到 `1000`。

预期：

- 下调和上调都不生成库存调整审核单。
- `inventory_sku_warehouse_stock.platform_total_qty` 直接变更。
- `inventory_stock_ledger` 写入普通库存调整流水。
- 库存调整审核列表无新增测试审核单。

通过标准：

- 不审核策略对退回和增加方向都真实放开。

## 6. 第三轮：强制审核策略测试

目标：验证 `ALWAYS` 真正强制进入审核，且方向范围有效。

配置和操作：

1. `ALWAYS + DECREASE`
   - 从 `1000` 下调到 `150`。
   - 预期生成审核单，不立即扣平台库存。
   - 从 `1000` 上调到 `1100`。
   - 预期不生成审核单，直接增加。
2. `ALWAYS + INCREASE`
   - 从 `1000` 下调到 `150`。
   - 预期不生成审核单，直接减少。
   - 从 `150` 上调到 `1000`。
   - 预期生成审核单，不立即增加。
3. `ALWAYS + BOTH`
   - 下调生成审核单。
   - 上调生成审核单。

通过标准：

- `ALWAYS` 不依赖销量阈值。
- `direction_scope` 对退回和增加方向均有效。
- 进入审核后，库存总览中的平台库存不立即变化。

## 7. 第四轮：条件审核阈值测试

目标：验证 `CONDITIONAL` 按销量窗口、保留天数和可立即退回数量生效。

测试前准备：

- 查询或构造测试 SKU 的 `inventory_sku_sales_daily` 近 7 天、近 30 天销量。
- 如果当前没有销量数据，可插入测试 SKU 的日销量聚合数据，并在测试记录中标明。

测试项：

1. `salesWindowDays=[7,30] + reserveDays=7`
   - 近 7 日日均和近 30 日日均取大值。
   - 申请退回数量大于可立即退回数量，应生成审核单。
2. `salesWindowDays=[30] + reserveDays=1`
   - 保留库存变小。
   - 同样的下调如果未超过可立即退回数量，应直接生效。
3. `salesWindowDays=[7] + reserveDays=30`
   - 保留库存变大。
   - 小幅下调也可能进入审核。
4. 设置 `minReturnQtyToReview`
   - 申请退回数量低于门槛，不审核。
   - 等于或高于门槛，继续按阈值判断。
5. 设置 `minReturnRatioToReview`
   - 申请退回比例低于门槛，不审核。
   - 等于或高于门槛，继续按阈值判断。

通过标准：

- 审核单里的以下字段与预期一致：
  - `requested_adjust_qty`
  - `sales_7d_qty`
  - `sales_7d_daily_avg`
  - `sales_30d_qty`
  - `sales_30d_daily_avg`
  - `threshold_daily_avg`
  - `protected_retained_qty`
  - `min_retained_qty`
  - `immediate_returnable_qty`
- 触发和不触发审核的结果符合公式。

## 8. 第五轮：卖家绑定优先级测试

目标：验证卖家绑定优先于全局绑定，绑定状态和策略状态真实有效。

测试项：

1. 全局策略为 `ALWAYS + BOTH`，测试卖家绑定 `DISABLED + BOTH`
   - 测试卖家调整库存应不审核。
2. 测试卖家绑定停用
   - 应回退全局 `ALWAYS + BOTH`，调整库存应进入审核。
3. 测试卖家绑定策略停用
   - 应回退全局策略。
4. 全局策略停用且测试卖家无启用绑定
   - 应无审核策略命中，不触发审核。
5. 两条卖家绑定不同优先级
   - 数字更小的优先级应先命中。

通过标准：

- 命中策略和审核结果与绑定优先级一致。
- 停用绑定和停用策略真实生效。

## 9. 第六轮：审核单处理测试

目标：验证审核单的处理动作可用。

测试项：

1. 立即生效
   - 生成审核单后点击立即生效。
   - 平台总库存按 `actualEffectQty` 扣减或增加。
   - 审核单状态变为 `EFFECTIVE`。
   - 写入库存流水。
   - 写入操作日志。
2. 驳回
   - 生成审核单后点击驳回。
   - 平台总库存不变化。
   - 审核单状态变为 `REJECTED`。
   - 写入操作日志。
3. 调整计划生效时间
   - 修改计划生效时间。
   - 审核单保持 `WAITING`。
   - 操作日志记录时间变化。
4. 人工生效开关
   - `manualEffectAllowed=N` 且计划时间未到，立即生效应失败。
   - 将计划时间调整到已到期后，应允许生效。

通过标准：

- 状态、库存数量、流水、日志都一致。

## 10. 第七轮：等待期内库存变化后的实际退回测试

目标：验证用户强调的核心逻辑：申请退回多少，不代表最终一定能退回多少。

场景：

1. 平台总库存为 `1000`。
2. 申请退回 `850`，进入审核，库存仍保持 `1000`。
3. 等待期内模拟销售或库存消耗，将平台总库存调整到较低值，例如 `300`。
4. 审核单生效。

预期：

- 实际生效数量为：

```text
actualEffectQty = min(850, currentPlatformTotalQty - currentReservedQty)
```

- 如果当前平台总库存为 `300` 且锁定为 `0`，实际只退回 `300`。
- `unfulfilledQty = 850 - 300 = 550`
- 生效后平台总库存为 `0`

通过标准：

- 审核单实际生效数量、未满足数量和库存最终值全部符合公式。

## 11. 第八轮：到期自动生效任务测试

目标：验证 Quartz 任务和 `autoEffectEnabled` 生效。

测试项：

1. `autoEffectEnabled=Y`
   - 生成审核单。
   - 将计划生效时间调整为当前时间之前。
   - 手动触发 `inventoryAdjustmentReviewTask.effectDueReviews` 或等待调度。
   - 预期审核单自动生效。
2. `autoEffectEnabled=N`
   - 同样生成已到期审核单。
   - 触发任务。
   - 预期审核单仍保持 `WAITING`。

通过标准：

- 到期任务只处理允许自动生效的审核单。

## 12. 第九轮：库存总览批量调整测试

目标：验证批量调整和单条调整一致。

测试项：

1. 批量中包含一个会进入审核的 SKU 和一个不会进入审核的 SKU。
2. 保存批量调整。

预期：

- 会进入审核的行生成审核单，不立即改库存。
- 不进入审核的行直接改库存并写流水。
- 批量结果里显示生成审核单数量。

通过标准：

- 批量预览、保存结果、库存最终值和审核单数量一致。

## 13. 第十轮：恢复和收尾验证

目标：恢复测试现场，保证系统处于可用状态。

收尾项：

1. 将测试库存行平台总库存恢复到测试前基线。
2. 将测试策略和绑定停用，或按确认范围删除。
3. 确认默认策略和原有全局绑定状态符合预期。
4. 查询测试审核单、库存流水和操作日志，形成最终测试证据。
5. 运行自动化回归：
   - 后端库存模块测试。
   - 前端类型检查。
   - 前端库存调整审核合同测试。
   - 三端 manifest 检查。
6. 同步 CodeGraph。

通过标准：

- 没有未处理的测试 `WAITING` 审核单，除非明确保留。
- 测试库存恢复或记录了无法恢复原因。
- 所有自动化回归通过。
