# 自动最优渠道流程只读干跑记录

记录时间：2026-06-12
执行方式：只读查询现有运行库数据，模拟“买家自动最优渠道”从 SKU 到客户渠道、系统渠道、上游试算能力的完整链路。
数据影响：仅执行 `SELECT` / `DESC`，没有执行 DDL、DML，没有修改数据库。
数据源：按 `application.yml` / `application-druid.yml` 与 `.env.local` 确认，当前后端激活 MySQL 为远端 `fenxiao` 库。

## 先纠正一个点

之前我说“自动最优可能跨多个仓库导致跨币种比较”，这个说法在你当前业务假设下不应该作为常规路径。

现有数据也支持你的判断：

- `product_spu_warehouse` 当前有仓库绑定的 SPU 数：25。
- 当前所有 SPU 的最大国家数：1。
- 当前所有 SPU 的最大币种数：1。
- 有多个仓库的 SPU 只有 `spu_id=45`、`spu_id=46`，都是 `CA012/US/USD` 与 `NY013/US/USD`。

所以自动最优可以跨同一商品的多个仓库做比较，但按当前业务规则，这些仓库应该是同国家、同币种。实现里仍然建议加一个兜底校验：如果同一订单候选仓库出现多个国家或多个币种，直接拒绝本次自动最优，提示配置异常，而不是做跨币种比较。

- 尺寸重量：`28.00 cm * 20.00 cm * 3.00 cm`，`0.42 kg`。

- 客户未指定仓库。
- 模式：自动最优。

## Step 2：匹配报价方案

实际数据：

| 表 | 结果 |
| --- | --- |
| `quote_scheme` | 2 条方案 |
| `quote_scheme_channel` | 6 条渠道配置 |
| `quote_scheme_warehouse` | 0 条仓库配置 |

严格按新规则跑：

- 新规则要求报价方案必须绑定且只能绑定一个仓库。
- 当前运行库 `quote_scheme_warehouse` 是空的。
- 所以严格模式下，买家 `3` + SKU `49` + 仓库 `NY013` 匹配不到任何有效计费报价方案，流程在这里停止。

按旧数据兼容跑：

- `scheme_id=1`，方案名 `测试1`，方案类型 `BILLING`。
- 费用来源 `EXTERNAL_ESTIMATE`。
- 币种 `USD`。
- 适用对象 `ALL_BUYERS`。
- 仓库范围仍是旧值 `ALL_WAREHOUSES`。
- 生效优先级 `1`。

判断：

- 当前库的数据还没有跟上我们刚改的“报价方案必须单仓库”规则。
- 后续如果要真实跑自动最优，必须先把报价方案重新编辑保存一次，或者做受控迁移，把方案仓库补进 `quote_scheme_warehouse`。

## Step 3：匹配客户渠道

在旧数据兼容模式下，以 `scheme_id=1` 继续模拟。

报价方案渠道：

| 客户渠道 | 名称 | 自动模式处理 |
| --- | --- | --- |
| `UPS-S` | UPS | 可参与自动 |
| `USPS` | USPS11 | 可参与自动 |
| `USPS-SF` | USPS-客户面单 | 客户上传面单，自动模式排除 |

客户渠道可见性：

- `UPS-S` 的买家范围模式是 `EXCLUDE`，当前只排除了 `buyer_id=36`，所以 `buyer_id=3` 可以用。
- `USPS` 的买家范围模式是 `ALL`，所以 `buyer_id=3` 可以用。
- `USPS-SF` 虽然可见，但因为 `label_upload_required=REQUIRED`，自动模式排除；手动模式可以出现。

判断：

- 客户渠道匹配规则是通的。
- 自动模式和手动模式的差异点应该只在“是否排除客户上传面单渠道”等自动化能力限制上，不应该绕开报价方案和买家可见性。

## Step 4：客户渠道必须反查可执行系统渠道

以仓库 `NY013` 继续过滤系统渠道：

| 客户渠道 | 映射系统渠道 | 履约模式 | `NY013` 仓库绑定 | 结果 |
| --- | --- | --- | --- | --- |
| `UPS-S` | `UPS` | `CARRIER_LABELING` | 有 | 可执行 |
| `USPS` | `USPS` | `CARRIER_LABELING` | 无 | 不可执行 |
| `USPS` | `USPS-Z` | `DIRECT_FULFILLMENT_WAREHOUSE` | 无 | 不可执行 |

判断：

- 自动最优候选不能只看报价方案里的客户渠道，还必须提前校验：这个客户渠道最终至少有一个系统渠道能在订单仓库发货。
- 对这个样本来说，自动模式最后只剩 `UPS-S -> UPS -> NY013` 一个可执行候选。
- `USPS-Z` 虽然是直推履约仓模式，但当前没有任何系统渠道仓库绑定，所以现在不能用于这个订单。

## Step 5：买家侧费用试算

按业务目标，这一步应该调用报价仓或外部费用试算能力，算出买家应付费用：

- 运费。
- 操作费。
- 增值费。
- 出库时必须产生的附加费用。

实际数据和代码现状：

- 计费方案 `测试1` 的费用来源是 `EXTERNAL_ESTIMATE`。
- 仓库 `NY013` 当前只有 `FULFILLMENT` 配对：`LX-NY013-3275A1E1 / NY11751 / 富东海外仓`。
- 仓库 `NY013` 没有 `QUOTE` 配对。
- 现有 `FeeEstimateServiceImpl` 还没有真正接外部试算，只会返回 `LINGXING_ESTIMATE_FIELDS_UNCONFIRMED`，拒绝伪造金额。

判断：

- 如果买家侧试算必须走 `QUOTE` 配对，那么样本会在这里失败，因为 `NY013` 没有报价仓配对。
- 如果允许临时复用履约仓试算接口，那么可以继续，但这个规则需要明确写进实现，不应该隐式混用。
- 当前代码还不是自动最优服务。它要求前端先传 `originWarehouseCode`、`quoteSchemeId`、`channelCodes`，只是在单方案下返回渠道试算壳子。

## Step 6：系统成本链路

假设买家侧最终选中 `UPS-S`，内部系统渠道链路为：

| 字段 | 值 |
| --- | --- |
| 客户渠道 | `UPS-S` |
| 系统渠道 | `UPS` |
| 系统渠道名称 | UPS渠道 |
| 履约模式 | `CARRIER_LABELING` |
| 绑定仓库 | `NY013` |
| 物流商连接 | `AGG56-进取顺利-CD9BCA6C` |
| 物流商外部渠道 | `UPS-Ground/JQ3` |

判断：

- 系统成本链路可以拼出来。
- 因为 `UPS` 是物流商打单模式，所以成本应该是：履约仓成本试算 + 物流商面单成本。
- 这一步还需要两个外部适配器：履约仓成本试算、物流商面单价格试算。

但是按当前系统渠道仓库绑定继续过滤：

- `CA012` 没有自动可执行客户渠道。

## 当前卡点

1. 运行库报价方案还没有仓库明细，`quote_scheme_warehouse=0`，严格新规则下自动最优第一步就会失败。
2. 现有 `FeeEstimateRequest` 还是“前端指定仓库 + 指定方案 + 指定渠道”的结构，不是“SKU + 目的地 + 可选仓库 -> 系统解析候选”的结构。
3. 客户侧外部费用试算未接通，当前代码明确拒绝伪造金额。
4. 仓库 `NY013` 没有 `QUOTE` 配对，如果买家侧费用必须走报价仓，会失败。
5. `USPS` / `USPS-Z` 没有系统渠道仓库绑定，虽然客户渠道可见，但订单不可执行。
6. 库存读模型里 `sku_id=49` 的库存行 `warehouse_code` 为空，后续如果订单要按库存仓库编码判断，需要修数据或补映射规则。

## 目标流程建议

后续自动最优应该按这个顺序实现：

1. 输入买家、SKU 行、目的地、可选指定仓库。
2. 解析所有 SKU 所属 SPU 的仓库集合。
3. 多 SKU 订单取仓库交集；交集为空时失败，因为一个订单只能一个仓库发货。
4. 如果客户指定仓库，则指定仓库必须在交集里；如果不指定，就用交集里的全部候选仓库。
5. 校验候选仓库的国家和币种必须一致；不一致直接失败并提示配置异常。
6. 匹配生效中的计费报价方案：方案类型 `BILLING`、买家范围命中、仓库命中、生效时间命中、状态启用。
7. 取报价方案里的客户渠道，再叠加客户渠道自身的状态、买家可见规则。
8. 自动模式排除客户上传面单渠道；手动模式保留。
9. 对每个客户渠道，必须能找到至少一个映射系统渠道，并且系统渠道必须绑定订单候选仓库；否则候选不可执行。
10. 外部费用试算前，先校验需要的上游配对、物流商映射是否存在；缺失就让该候选失败，不调用 API。
11. 对剩余候选调用外部试算；单候选失败只淘汰该候选，全部失败才让本次自动最优失败。
12. 客户侧按总应付费用选最低；费用相同再按匹配精确度、方案优先级、渠道排序、编码兜底排序。
13. 客户渠道确定后，再在映射系统渠道中计算系统成本，按总成本最低选择最终系统渠道。
14. 保存费用快照和候选失败摘要，供订单后续使用和排障。

## 性能判断

性能上不能按“仓库 * 客户渠道 * 系统渠道 * API”直接暴力打外部接口。实现时应该先用 SQL 和内存规则把候选压缩到最小：

- 仓库集合来自 SKU/SPU 仓库交集，不扫全部仓库。
- 报价方案必须先按买家、仓库、生效时间、类型、状态过滤。
- 客户渠道必须先按报价方案、买家可见性、自动能力过滤。
- 系统渠道必须先按客户渠道映射和仓库绑定过滤。
- 上游配对、物流商映射缺失的候选直接失败，不进入外部 API。
- 外部调用做请求内去重、短缓存、并发上限和候选数量上限。

按这条链路，正常一单不会真的扩散到大量 API 调用；真正有风险的是配置过宽或缺少前置过滤。

## 结论

业务流程方向是通的，但当前运行库和现有费用试算服务还没有完全具备自动最优的运行条件。

现在最关键的下一步不是先写 API 调用，而是先实现“候选解析器”：只根据买家、SKU、目的地、可选仓库，把候选仓库、报价方案、客户渠道、系统渠道、上游配对检查结果一次性算出来，并把失败原因列清楚。这个解析器跑通以后，再接外部试算和成本择优会稳很多。
