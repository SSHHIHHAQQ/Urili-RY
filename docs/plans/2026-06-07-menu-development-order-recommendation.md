# 菜单开发顺序建议

## 目标

根据当前运行库 `sys_menu` 和本地前端页面文件，重新整理后续菜单开发顺序，避免直接跳到订单，也避免把审核中心当成后置的独立 CRUD 菜单。

## 数据来源

- 运行库：当前后端 `.env.local` 指向远程 MySQL `fenxiao`，本次只读查询 `sys_menu`，未写入数据。
- 前端：检查 `react-ui/src/pages` 下对应组件文件是否存在。
- 判断规则：
  - `Common/PlannedPage/index`：占位，未做真实业务。
  - 组件文件存在：已有真实页面或至少已有页面骨架。
  - 组件文件缺失：菜单已存在，但页面还没有落地。

## 推荐方向

我的建议不是按菜单目录从上到下做，而是按业务依赖链做：

```text
审核基座
  -> 商品发布审核闭环
  -> 渠道/计费/报价基底
  -> 平台库存账本
  -> 资金账户基底
  -> 订单最小闭环
  -> 退件/售后
  -> 财务对账和利润
```

核心判断：

- 订单不能现在直接做，因为订单会同时依赖商品审核结果、卖家/买家主体、仓库、渠道、报价、库存锁定、费用、资金账户。
- 审核中心不能最后做，因为商品新建后的审核是商品发布链路的一部分。
- 但审核中心也不能做成单独的“大审核 CRUD”，应该先做审核基座，再把商品分销审核作为第一个落地场景。

## 推荐开发顺序

### 1. 先做审核基座，不先做审核中心页面大全

优先级：P0

先做一个可复用的审核模型和审核流：

- 审核对象：业务类型、业务单号、发起人、发起时间、当前状态。
- 审核动作：提交审核、通过、驳回、撤回、重新提交。
- 审核记录：每次审核意见、操作人、操作时间、结果。
- 状态约束：不能随便从任意状态跳任意状态。

第一版只服务商品，不要一开始抽成过大的万能流程引擎。

建议落地方式：

- 菜单上先只做 `审核中心 / 商品分销审核`。
- 商品页面里增加“提交审核”“审核状态”“审核记录”。
- 审核中心只是运营处理待审商品的工作台，不是商品数据的第二套维护入口。

### 2. 做商品分销审核闭环

优先级：P0

原因：

- 你说得对，商品新建后必须审核，这一步应该在订单之前。
- `商城商品列表` 已经是商品主入口，审核应该回写商品状态，而不是新建一套孤立审核数据。

建议顺序：

1. 商品保存草稿。
2. 商品提交审核。
3. `商品分销审核` 列出待审商品。
4. 审核通过后商品进入可上架/可售状态。
5. 审核驳回后回到可编辑/待重新提交状态。
6. 商品操作日志记录审核前后状态。

这一步完成后，商品才算具备进入订单链路的资格。

### 3. 补渠道管理

优先级：P1

当前 `渠道管理 / 系统渠道管理`、`客户渠道管理` 菜单存在，但页面缺失。

建议先做渠道，是因为后续海外仓报价、订单履约、退件、费用结算都要用渠道。尤其你现在上游系统里已经有物流渠道同步和渠道配对，如果系统渠道本身没稳定，订单后面会很乱。

建议顺序：

1. 系统渠道管理。
2. 客户渠道管理。
3. 上游渠道配对回到上游系统页面里使用。

### 4. 补海外仓服务设置里的计费和报价基底

优先级：P1

当前 `操作费设置`、`运费设置`、`报价方案` 菜单存在，但页面缺失。

建议顺序：

1. 操作费设置。
2. 运费设置。
3. 报价方案。

原因：

- 订单之前必须有费用规则，否则订单金额、履约报价、利润对账都没有可信来源。
- 报价方案最好依赖渠道、仓库、费用规则，不要先孤立做。

### 5. 做平台库存账本，而不是只看来源仓库库存

优先级：P1

当前 `来源仓库库存` 已有页面，但它只是上游库存读模型，不等于平台真实库存。

建议下一步不要直接做 `订单列表`，而是先做：

1. `库存总览`：平台维度的 SKU + 仓库库存余额。
2. `库存流水`：只追加记录库存变化。
3. `库存调整审核`：作为库存调整动作的审核场景。

注意：

- 来源仓库库存可以作为同步参考。
- 平台库存账本才是订单锁定、发货扣减、退件回补的依据。
- 库存调整审核应该在库存账本有了以后做，不应该现在独立做。

### 6. 做财务资金基底

优先级：P2

当前 `资金账户`、`收款账户`、`分销资金` 都是占位。

建议顺序：

1. 收款账户。
2. 资金账户。
3. 分销资金。
4. 充值审核。

原因：

- `充值审核` 不应该先做，它必须依赖充值申请、资金账户和收款账户。
- 没有资金账户模型，充值审核会变成无来源、无去向的孤立审核。

### 7. 再做订单列表的最小闭环

优先级：P2

等商品审核、渠道、报价、库存、资金基底都有后，再做 `订单列表`。

首版订单不建议一步到位做完整订单系统，应该只做最小闭环：

1. 订单来源和订单主体。
2. 商品 SKU 明细。
3. 买家/卖家主体。
4. 仓库和渠道。
5. 金额快照。
6. 库存锁定状态。
7. 订单状态流转。

第一版不要急着把退件、售后、利润都塞进订单列表。

### 8. 最后做退件管理和售后管理

优先级：P3

`退件管理` 和 `售后管理` 都依赖订单、库存、费用、财务。

建议顺序：

1. 售后管理：先定义售后单和订单关系。
2. 退件管理：再定义实物退回、仓库入库、库存回补。

不要把退件直接做成订单的一个字段，它需要自己的过程和记录。

### 9. 财务费用和利润对账后置

优先级：P3

`费用管理` 和 `利润对账` 不建议现在做完整功能。

原因：

- 利润依赖订单金额、渠道运费、操作费、仓储费、售后退款、库存损耗。
- 如果现在先做利润对账，会只能做假数据或手填数据，后面很容易返工。

建议：

- `费用管理` 可先做费用项字典/规则入口。
- `利润对账` 等订单和费用快照稳定后再做。

## 按组件和板块拆分

### 审核组件

建议沉淀：

- `ReviewStatusTag`
- `ReviewTimeline`
- `ReviewActionModal`
- `ReviewReasonForm`

先服务商品审核，后续复用到库存调整和充值审核。

### 商品组件

建议沉淀：

- 商品基础信息表单。
- SKU 编辑表格。
- 属性配置选择器。
- 商品仓库绑定组件。
- 商品审核状态区块。

商品分类和属性不要再继续扩大成独立业务中心，应作为商品编辑和筛选的配置来源。

### 渠道和报价组件

建议沉淀：

- 渠道选择器。
- 仓库 + 渠道组合选择器。
- 费用规则表格。
- 报价方案明细组件。

这些组件会被报价、订单、售后、利润复用。

### 库存组件

建议沉淀：

- SKU + 仓库库存查询条件。
- 库存余额卡片/表格。
- 库存流水表格。
- 库存调整申请弹窗。

库存页面要严格区分“来源库存读模型”和“平台真实库存账本”。

### 财务组件

建议沉淀：

- 金额展示组件。
- 币种选择器。
- 账户选择器。
- 资金流水表格。

金额和流水必须为后续只追加记录留接口，不要做覆盖式 CRUD。

## 最推荐的下一步

如果现在只选一个方向，我建议：

```text
先做 商品分销审核闭环
```

但它不要从 `审核中心` 页面孤立开工，而应该从这条链路开工：

```text
商城商品列表
  -> 商品提交审核
  -> 商品分销审核
  -> 审核通过/驳回
  -> 商品状态回写
  -> 商品操作日志
```

这样既满足“审核要提前”，又不会把审核中心做成一个和业务割裂的菜单。

## 暂不建议立刻做

- `订单列表`：依赖太多，现在做会用占位字段堆起来，后面返工概率高。
- `退件管理`：必须等订单和库存账本有了。
- `售后管理`：必须等订单、费用、退款、库存回补规则有了。
- `利润对账`：必须等订单金额、费用快照、售后结果稳定。
- `充值审核`：必须等资金账户、收款账户、充值申请来源有了。
