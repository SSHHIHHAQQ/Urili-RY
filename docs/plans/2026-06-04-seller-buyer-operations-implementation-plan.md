# 卖家/买家管理补齐操作实施确认方案

日期：2026-06-04

本方案覆盖以下功能：

- 账号重置密码
- 免密登录端
- 买家充值
- 余额展示/筛选
- 创建/最后登录时间筛选

本方案是实施前确认文档，不是已落地实现。

## 2026-06-04 用户确认后的范围调整

- 卖家分销账户余额：当前要展示，但只做占位，后续再补真实来源。
- 买家充值：当前只需要占位字段，不做充值功能、不做弹窗、不做充值接口。
- 卖家端、买家端 URL：可以先用若依参数配置占位。
- 免密登录 token 有效期：30 分钟。

## 总体原则

- 仍以若依架构为核心：用户、角色、菜单、权限、字典、操作日志优先复用若依。
- 卖家和买家主体继续保持独立模块、独立表，不回到通用 customer 表。
- 账号继续复用若依 `sys_user`，通过 `seller_account`、`buyer_account` 绑定到各自主体。
- 充值、余额、流水不写进 `buyer` 主体资料表；必须进入财务/余额模型。
- 金额使用 `BigDecimal` / `decimal`，不使用 `float`、`double`。
- 充值流水只追加，不直接覆盖历史。

## 推荐默认方案

### 1. 账号重置密码

推荐做法：

- 保留当前账号弹窗里的账号列表。
- 在账号列表每一行增加“重置为默认密码”操作。
- 默认密码使用当前已确认的 `U12346`。
- 管理端主表可以增加“重置主账号密码”快捷操作，作用对象是该客户的 `OWNER` 账号。
- 每次重置写若依操作日志。

不建议：

- 不建议只做“客户级重置密码”，因为现在一个客户会有主账号和多个子账号，客户级操作不够明确。
- 不建议让前端随便输入新密码作为唯一入口；这和“默认密码”规则不一致，也不利于管理端统一操作。

需要确认：

1. 主表行操作是否也要显示“重置主账号密码”？推荐显示。
2. 子账号列表里是否允许重置任意子账号？推荐允许，受 `seller:admin:resetPwd` / `buyer:admin:resetPwd` 权限控制。

### 2. 免密登录端

推荐做法：

- 管理端卖家列表增加“登录卖家端”。
- 管理端买家列表增加“登录买家端”。
- 默认登录该客户的 `OWNER` 主账号。
- 账号弹窗里可以后续扩展“以此账号登录”，但第一版先只做主账号，降低风险。
- 后端生成一次性 token：
- 短有效期，本轮按已确认的 30 分钟。
  - 只能消费一次。
  - 禁止停用的卖家/买家、停用的账号、停用的若依用户登录。
  - 写操作日志，记录管理员、客户、目标账号、过期时间。
- 登录链接中的前端地址从若依参数配置读取：
  - `portal.seller.web.url`
  - `portal.buyer.web.url`

不建议：

- 不建议照搬原项目的 in-memory token，因为后端重启会丢，且不适合远程环境。
- 不建议前端自行拼登录态或绕过后端权限。

需要确认：

1. 免密登录默认是否只登录 `OWNER` 主账号？推荐是。
2. 卖家端、买家端当前 URL 是否先用配置占位？如果三端前端还没拆出来，可以先写入参数配置，后续替换。
3. token 有效期已确认按 30 分钟。

### 3. 买家充值

用户已确认：

- 当前只做占位字段。
- 不做充值功能。
- 不做充值弹窗。
- 不做充值接口。
- 不新增钱包表或充值流水表。

推荐当前落地方式：

- 买家列表保留“充值占位”展示字段或占位列，明确显示为“待接入”。
- 业务代码不产生金额变动。
- 后续需要真正充值时，再单独提交财务/余额流水设计方案。

不建议：

- 不建议把余额字段直接加在 `buyer` 表上，然后每次充值直接 update。
- 不建议第一版做复杂财务全模块，否则会拖慢当前管理端验证。

需要确认：

1. 默认币种是否使用 `USD`？原项目余额使用 USD，推荐第一版默认 USD。
2. 充值是否“管理员提交后立即入账”？推荐第一版立即入账。
3. 当前不进入充值规则设计；后续做真实充值时再确认。

### 4. 余额展示/筛选

推荐做法：

- 卖家列表展示“分销账户余额”占位。
- 买家列表展示“账户余额”占位。
- 卖家/买家列表增加余额最小值、最大值筛选占位。
- 当前余额值统一返回 0，不代表真实余额。
- 后续真实余额来源明确后，再替换为财务/结算聚合。

待确认点：

- 原项目卖家列表也有“分销账户余额”。当前是否也要在卖家列表展示余额并筛选？

推荐分两步：

1. 本轮卖家、买家都展示余额占位。
2. 本轮不做真实充值、不做真实余额流水。
3. 卖家分销账户余额、买家账户余额的真实来源后续分别按财务/结算规则补。

### 5. 创建/最后登录时间筛选

推荐做法：

- 卖家列表、买家列表都增加：
  - 创建时间范围筛选
  - 最后登录时间范围筛选
- 后端 Mapper 增加 `create_time`、主账号 `login_date` 范围条件。
- 前端 ProTable 默认展开状态继续沿用当前已做的浏览器保存逻辑。

这个功能不涉及新业务表，风险低，可以先做。

## 权限点建议

卖家：

- `seller:admin:resetPwd`
- `seller:admin:directLogin`

买家：

- `buyer:admin:resetPwd`
- `buyer:admin:directLogin`
- `buyer:admin:rechargePlaceholder`

已有 `seller:admin:resetPwd`、`buyer:admin:resetPwd` 可复用；directLogin 需要新增菜单按钮权限。买家充值当前只是占位字段，不需要后端充值接口；如果前端展示占位列，可不新增按钮权限。

## 推荐实施顺序

1. 只读确认当前远程库：
   - `sys_role` 是否有 `seller`、`buyer`。
   - 菜单权限是否有当前卖家/买家权限点。
   - 字典是否有 `country_region`、`subject_type`、`seller_level`、`buyer_level`、账号角色。
2. 先补时间筛选和默认密码重置。
3. 实现免密登录 token 生成、接口、权限和前端入口，token 有效期 30 分钟。
4. 补卖家/买家余额占位展示和筛选。
5. 补买家充值占位字段，不做充值弹窗和接口。
6. 运行后端编译、前端构建，并用浏览器验证管理端操作入口。

## 必须确认的问题

如果没有额外要求，建议按以下默认值推进：

1. 重置密码：主表重置 `OWNER` 主账号；账号弹窗可重置任意子账号；默认密码 `U12346`。
2. 免密登录：主表只登录 `OWNER` 主账号；token 有效期 30 分钟；禁用状态一律不允许登录。
3. 三端 URL：先放入若依参数配置，卖家端/买家端前端拆出来后替换。
4. 买家充值：本轮只做占位字段，不做功能、不做弹窗。
5. 余额展示：本轮卖家、买家都做占位展示/筛选，真实余额后续接财务/结算来源。
