# 卖家/买家管理完整整改计划

日期：2026-06-04

## 目标

本计划用于整改当前若依验证工程里的卖家/买家管理能力。范围不只包含用户提到的菜单和字段，也覆盖数据模型、运行库一致性、附件存储、账号体系、权限、字典、前端结构、财务边界、文档和验证流程。

当前计划只定义整改路线和验收标准，不直接执行数据库 DDL/DML，不提交真实环境写入操作。涉及远端数据库菜单、权限、字典、业务表和历史数据迁移时，必须另写执行记录并得到确认后再执行。

## 当前边界

- 后端仍以 `RuoYi-Vue/` 为唯一若依后端基线。
- 当前阶段前端仍以 `react-ui/` 作为管理端验证入口，不立即复制出三套前端。
- 卖家和买家继续按独立模块、独立表、独立权限处理，不回退到通用 customer 单表。
- 账号继续复用若依 `sys_user`、`sys_role`、`sys_menu`、`sys_dict` 和操作日志能力，不另起一套账号/权限/字典体系。
- COS 文件存储已经接入 `FileStorageService`，后续业务附件应迁移到统一文件服务，不再新增 data URL 入库。

## 已知现状

### 已落地基础

- 后端已有 `seller`、`buyer` 两个业务模块。
- SQL 种子中已有 `seller`、`buyer`、`seller_account`、`buyer_account` 方向。
- 前端已有 `react-ui/src/pages/Seller/` 和 `react-ui/src/pages/Buyer/` 入口。
- 管理端共用 `PartnerManagementPage` 承载卖家/买家主体资料、账号弹窗、附件控件和字典转换。
- 复用台账已经登记主体字典、等级字典、账号角色字典、国家/地区字典、账号支撑类、直登支撑类和文件存储服务。
- 文件存储已支持本地与腾讯 COS，真实 COS 上传已通过 `/common/upload` 验证。

### 主要问题

- 菜单和路由仍存在历史 customer / Urili 命名残留，容易误导后续开发。
- 当前附件仍由前端读取为 data URL 后写入业务表 `attachment_file_url`，不适合正式业务附件。
- `PartnerManagementPage.tsx` 已超过 500 行，继续加功能会放大维护风险。
- 运行库是否完整执行当前 seller/buyer 菜单、角色、权限、字典、表结构种子，需要只读核查。
- 余额、充值、分销账户余额等属于财务/结算事实，不应直接塞进卖家/买家主体表。
- 免密登录、默认密码重置、账号禁用联动、操作审计需要按三端账号模型统一。
- 历史文档和种子文件中仍有旧 customer 单表方案痕迹，需要归档或标注为历史方案。

## 整改原则

- 先稳定管理端，再推进卖家端、买家端物理拆分。
- 先只读核查运行库，再决定是否执行菜单、权限、字典或表结构写入。
- 新增业务能力必须先判断归属模块，不能靠 `type` / `kind` 长期混放不同主体。
- API 和数据库保存 code，前端通过字典或统一 option catalog 展示 label。
- 后端权限必须 fail closed，不能只靠前端隐藏按钮。
- 附件、图片、凭证、导入文件统一走 `FileStorageService`，业务表只保存资源路径或文件标识。
- 财务、库存、外部请求日志必须可追溯，不用简单 CRUD 覆盖核心状态。
- 每个整改阶段都要留下 Markdown 记录，说明目标环境、影响范围、验证命令和未验证原因。

## 整改范围

### 1. 菜单、路由和命名

目标：消除会影响后续理解和权限配置的旧 customer / Urili 包装命名。

整改项：

- 将管理端根菜单口径统一为“主体管理”或“卖家/买家管理”，默认推荐“主体管理”。
- 在根菜单下明确二级菜单：
  - 卖家管理，对应 `seller`。
  - 买家管理，对应 `buyer`。
- 路由、页面、权限点继续使用业务对象名：
  - `/seller/...`
  - `/buyer/...`
  - `seller:admin:*`
  - `buyer:admin:*`
- 清理仍在有效种子、菜单字段记录、前端 route name 中出现的旧 `customer`、`UriliCustomer`、`UriliProduct` 这类包装命名。
- 历史草稿文档不强行删除，但要标注为历史方案，不能作为当前实现依据。

验收标准：

- 有效 SQL 种子中不再把卖家/买家菜单挂在“客户管理”下。
- 当前有效路由和权限不再需要通过 `customer_kind` 或通用 customer 语义区分卖家/买家。
- `docs/menu-fields.md` 与实际菜单种子保持一致。

### 2. 运行库一致性核查

目标：确认远端运行库和当前代码/种子是否一致，避免代码正确但运行环境缺角色、权限、字典或表。

整改项：

- 执行前先读取当前激活配置：
  - `RuoYi-Vue/ruoyi-admin/src/main/resources/application.yml`
  - `RuoYi-Vue/ruoyi-admin/src/main/resources/application-druid.yml`
  - 本机 `.env.local` 仅用于读取连接来源，不在报告中输出明文。
- 只读核查 MySQL：
  - `seller`、`buyer`、`seller_account`、`buyer_account` 是否存在。
  - 是否残留旧 `urili_customer` / `urili_customer_account` 运行表。
  - `sys_role.role_key` 是否存在 `seller`、`buyer`。
  - `sys_menu.perms` 是否存在当前后端需要的权限点。
  - `sys_dict_type` / `sys_dict_data` 是否存在主体类型、等级、账号角色、国家/地区字典。
- 只读核查 Redis：
  - 确认后端实际使用远端还是本地 Redis。
  - 确认直登 token 前缀不会和其他业务 key 冲突。

验收标准：

- 输出 Markdown 核查记录，写清连接来源、目标环境、命令类型和是否只读。
- 未经确认不执行任何 DDL/DML。
- 若发现本地 Docker MySQL/Redis 正在运行，记录其可能造成误判，但默认不读取它。

### 3. 附件和文件存储

目标：把卖家/买家附件从 data URL 入库整改为统一文件服务路径。

当前问题：

- 前端 `Upload` 使用 `beforeUpload={() => false}` 阻止自动上传。
- 文件被浏览器读取为 data URL。
- 后端业务表保存 `attachment_file_name`、`attachment_mime_type`、`attachment_size_bytes`、`attachment_file_url`。
- `attachment_file_url` 当前可能保存长 base64 字符串，不利于数据库体积、缓存、接口返回和安全控制。

整改项：

- 前端选择附件后调用 `/common/upload`。
- `/common/upload` 底层使用 `FileStorageService`，根据环境写入本地或 COS。
- 表单 payload 只保存：
  - 文件名
  - MIME 类型
  - 文件大小
  - `/profile/...` 资源路径或后续文件 ID
- 禁止新增 data URL 写入业务表。
- 列表和详情展示附件时，通过后端资源路径访问，不由前端拼接 COS endpoint。
- 增加附件类型和大小校验：
  - 图片
  - PDF
  - Word / Excel
  - CSV / TXT
  - 默认单文件，超限拒绝并给出明确提示。
- 历史 data URL 处理分两步：
  - 第一阶段保留兼容读取，避免旧数据立即不可用。
  - 第二阶段另写迁移方案，将 data URL 转为文件服务对象并回填资源路径。

是否需要新增文件表：

- 当前卖家/买家单附件可以先沿用现有字段。
- 如果后续订单凭证、财务凭证、面单、批量导入文件都复用附件能力，再单独设计 `file_resource` 或业务附件表。
- 新增文件表属于建表事项，必须另写表设计方案并确认。

验收标准：

- 新上传卖家/买家附件不再产生 data URL。
- COS 模式下上传后能通过后端返回路径访问。
- 数据库中新增附件路径以 `/profile/...` 或确认后的文件 ID 保存。
- 真实 COS 凭证只存在本机环境变量或 `.env.local`，不进入代码、报告和日志。

### 4. 卖家/买家主体字段

目标：补齐管理端需要的字段，但不把财务和业务事实混入主体资料表。

主体资料应承载：

- 内部编号
- 卖家/买家代码
- 名称
- 简称
- 主体类型
- 等级
- 联系人
- 电话
- 邮箱
- 国家/地区
- 州/省
- 城市
- 邮编
- 地址
- 附件元数据
- 备注
- 状态
- 创建时间
- 更新时间

不应承载：

- 真实余额
- 充值流水
- 财务结算流水
- 库存事实
- 订单事实
- 外部请求日志

整改项：

- 确认当前前后端字段与表结构、Mapper、Domain、TypeScript 类型一致。
- 统一字段 code / label 规则：
  - 表和接口保存 code。
  - 页面展示 label。
  - 选项来自若依字典或 option catalog。
- 对创建时间、最后登录时间开放筛选：
  - 主体创建时间来自 seller/buyer 表。
  - 最后登录时间来自 OWNER 账号对应 `sys_user.login_date`。
- 对余额类字段只做占位展示时，明确标注为待接入，不参与真实财务计算。

验收标准：

- 表单、列表、详情和接口字段命名一致。
- 无新增内联大段下拉选项或颜色映射。
- 余额/充值没有通过主体表直接 update 真实金额。

### 5. 账号体系和免密登录

目标：让管理端、卖家端、买家端账号模型清晰，避免一个若依用户跨端混用。

整改项：

- 继续复用 `sys_user`。
- 继续通过 `seller_account`、`buyer_account` 绑定到各自主体。
- Service 层保持同一个 `sys_user` 不能同时绑定卖家和买家的校验。
- 只读核查远端库是否已有跨表重复绑定脏数据。
- 密码重置规则统一：
  - 保留账号弹窗内“重置指定账号”。
  - 主列表增加“重置 OWNER 主账号默认密码”时，必须明确作用对象。
  - 默认密码使用已确认值，并由后端常量统一维护。
- 启停规则统一：
  - 停用主体时，评估是否同步停用 OWNER 和子账号。
  - 停用账号、停用 `sys_user`、停用主体都不得允许免密登录。
- 免密登录规则：
  - 管理端调用后端生成一次性 token。
  - token 写入 Redis。
  - 默认有效期 30 分钟。
  - token 只能消费一次。
  - 管理端只打开配置化的卖家端/买家端 URL，不在前端伪造登录状态。
  - 写操作日志，记录管理员、主体、目标账号、过期时间和结果。

验收标准：

- 无权限用户不能看到或调用直登、重置密码按钮。
- 停用主体、停用账号、停用若依用户都不能直登。
- Redis token 可过期、可一次性消费。
- 直登 URL 来自若依参数配置，不写死到页面。

### 6. 权限和审计

目标：管理端按钮、后端接口、菜单种子和操作日志一致。

整改项：

- 现有权限点复核：
  - `seller:admin:list`
  - `seller:admin:query`
  - `seller:admin:add`
  - `seller:admin:edit`
  - `seller:admin:changeStatus`
  - `seller:admin:resetPwd`
  - `buyer:admin:list`
  - `buyer:admin:query`
  - `buyer:admin:add`
  - `buyer:admin:edit`
  - `buyer:admin:changeStatus`
  - `buyer:admin:resetPwd`
- 待新增或复核权限点：
  - `seller:admin:directLogin`
  - `buyer:admin:directLogin`
  - 后续真实充值再设计 `buyer:finance:recharge` 或财务模块权限，不在当前主体管理中抢先定义。
- 后端新增接口必须配置 `@PreAuthorize`。
- 有状态变更、重置密码、直登 token 生成、附件更新的接口必须配置 `@Log` 或等价审计记录。
- 前端按钮使用权限控制，但不能替代后端权限。

验收标准：

- 管理员有权限时功能可用。
- 移除权限后前端按钮不可见，直接请求接口也被后端拒绝。
- 操作日志能定位操作者、目标主体、目标账号和动作结果。

### 7. 字典、选项和展示格式

目标：避免每个页面各写一套状态、等级、国家、账号角色和展示颜色。

整改项：

- 复核并补齐字典：
  - `subject_type`
  - `seller_level`
  - `buyer_level`
  - `seller_account_role`
  - `buyer_account_role`
  - `country_region`
- 前端统一使用字典 hook 或 option catalog。
- 页面不内联大段国家/地区列表。
- 状态、等级、账号角色、主体类型保存 code，展示 label。
- 新增币种、余额类型、附件分类等字典前，先确认是否已进入财务或文件模块设计。

验收标准：

- 复用台账记录所有新增字典、option catalog、formatter。
- 前端 mock 或占位数据使用真实 code，不另造展示 label。
- 类型检查能覆盖新增 option 的 code 范围。

### 8. 前端结构整改

目标：控制 `PartnerManagementPage` 继续膨胀，避免把表格、表单、账号弹窗、附件上传和业务规则继续堆在一个文件。

整改项：

- 拆分建议：
  - `PartnerManagementTable`
  - `PartnerProfileForm`
  - `PartnerAccountModal`
  - `PartnerAttachmentUpload`
  - `usePartnerDictionaries`
  - `usePartnerAccounts`
  - `partnerColumns`
  - `partnerFormSchema`
- 保持 `pages/Seller` 和 `pages/Buyer` 只作为入口，不把业务逻辑写回页面入口。
- 前端 API 只通过 `services/seller`、`services/buyer` 调用后端。
- 移除或隔离验证阶段 mock，明确 mock 与真实接口边界。
- 新增上传、直登、密码重置、筛选项时优先先拆结构，再加功能。

验收标准：

- `PartnerManagementPage.tsx` 不继续承载新增功能。
- 拆分后每个文件职责单一。
- 表格列、表单项、上传控件和账号弹窗可被卖家/买家复用，但业务入口仍独立。
- `npm run tsc` 通过。

### 9. 财务、余额和充值

目标：当前只允许做占位展示，不把财务事实写进主体资料。

当前决策：

- 卖家分销账户余额：当前可以展示占位。
- 买家账户余额：当前可以展示占位。
- 买家充值：当前只保留占位入口或待接入状态，不做真实充值接口、弹窗和流水。

后续真实实现前必须另写表设计，至少说明：

- 账户模型
- 币种
- 金额精度
- 入账/出账/冻结/解冻
- 充值流水
- 冲销/作废
- 审核规则
- 操作人和原因
- 与订单、结算、退款的关系
- 权限点
- 审计日志
- 回滚方式

验收标准：

- 当前卖家/买家管理不产生真实金额变动。
- 任何真实充值或余额更新都必须进入财务模块设计确认流程。
- 金额字段一律使用 `BigDecimal` / `decimal`，不使用 `float` / `double`。

### 10. 历史数据和迁移

目标：避免历史 customer 方案、data URL 附件和远端库状态不明带来二次污染。

整改项：

- 只读核查是否存在旧 customer 表或旧菜单。
- 如果存在旧表，先输出迁移对照：
  - 旧客户类型到 seller/buyer 的拆分规则。
  - customer code 到 seller_code / buyer_code 的映射。
  - 账号绑定迁移规则。
  - 附件 data URL 到文件服务对象的迁移规则。
  - 菜单、角色、权限迁移规则。
- 迁移前必须有备份、回滚和抽样校验方案。
- 迁移脚本必须可重复审查，不直接在交互里临时拼 SQL。

验收标准：

- 迁移前后记录行数、异常行、跳过原因。
- 迁移后不留下同一主体同时存在旧 customer 和新 seller/buyer 的可写路径。
- data URL 迁移失败的记录可追踪、可重试。

### 11. 外部系统和后续业务模块

目标：避免卖家/买家主体管理变成所有业务事实的容器。

边界：

- 商品进入 product。
- 库存进入 inventory。
- 订单进入 order。
- 履约进入 fulfillment。
- 财务进入 finance。
- 领星、WMS、ERP、物流、支付进入 integration 或明确适配层。

整改项：

- 卖家/买家主体表只存主体资料和管理状态。
- 外部系统账号、请求日志、库存流水、财务流水不写入主体资料表。
- 外部适配器不得绕过业务模块直接写事实表。
- 外部请求日志只追加，必须记录 traceId、请求时间、响应时间、外部单号、本地业务单号、错误码和脱敏错误信息。

验收标准：

- 新需求先判断模块归属。
- 跨模块调用通过 Service、Facade 或明确公共接口。
- 不通过一个宽泛表加 `biz_type` 长期混放不同业务事实。
