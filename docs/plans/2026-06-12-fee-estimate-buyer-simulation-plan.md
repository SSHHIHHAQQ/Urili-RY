# 费用试算买家模拟视图计划

日期：2026-06-12

## 背景

当前 `费用试算` 已经作为管理端财务工具落在 `财务管理` 下，用于运营人员选择发货仓、报价方案、物流渠道、包裹信息后，验证报价方案和费用试算链路。

现在需要在同一个页面内增加一个更贴近买家下单过程的视图。这个视图仍然是管理端工具，不是买家端正式页面；它用于让管理端模拟“某个买家选择商品后，系统如何选仓、选客户渠道、解析系统渠道，并分别得到买家费用和系统成本”。

## 目标

1. 在费用试算左侧条件面板顶部增加视图 Tabs。
2. 保留现有管理端试算能力，不破坏当前手工尺寸和选择 SKU 的试算流程。
3. 新增 `买家模拟` 视图，按买家下单逻辑组织输入项和结果看板。
4. 买家模拟支持两种仓库/渠道选择方式：
   - `手动指定`：手动选择仓库，再选择客户渠道。
   - `自动最优`：系统自动选择最优仓库和客户渠道，可选限制仓库范围。
5. 右侧上方展示候选解析过程，右侧下方展示最终试算结果。
6. 第一阶段不新增业务表，不保存试算历史，不写订单费用快照。

## 页面视图划分

左侧条件面板顶部增加 Tabs：

| Tab | 建议名称 | 用途 |
| --- | --- | --- |
| 管理端视图 | `运营试算` | 当前已经实现的管理端试算工具。运营指定发货仓、报价方案、物流渠道和包裹信息。 |
| 买家视图 | `买家模拟` | 管理端模拟买家下单视角。选择买家和商品后，由系统解析可用仓库、客户渠道、系统渠道和费用。 |

`运营试算` 继续保留现有布局：

- 左侧条件面板。
- SKU 模式右侧为 `包裹信息 + 试算结果` 上下两块。
- 手工尺寸模式右侧只保留 `试算结果`，并铺满右侧区域。

`买家模拟` 使用：

- 左侧条件面板。
- 右上 `候选解析` 看板。
- 右下 `试算结果` 看板。

## 买家模拟左侧表单

### 字段顺序

1. 买家
2. 商品/SKU
3. 仓库/渠道选择方式
4. 根据选择方式展示仓库和渠道字段
5. 到货国家/地区
6. 到货省州
7. 到货城市
8. 到货邮编
9. 收件地址1
10. 收件地址2
11. 计算按钮

### 仓库/渠道选择方式

字段名称：`仓库/渠道选择方式`

选项：

| 选项 | 页面行为 | 业务含义 |
| --- | --- | --- |
| `手动指定` | 展示 `选择仓库` 和 `选择客户渠道` | 模拟买家明确选择一个仓库和一个客户渠道。 |
| `自动最优` | 展示 `限制仓库`，不展示渠道字段 | 系统自动在可用仓库和客户渠道里找最优方案。 |

### 手动指定模式

字段：

- `选择仓库`：单选，必填。
- `选择客户渠道`：单选，必填。

说明：

- 这个模式模拟买家已经明确指定仓库和渠道。
- 如果需要一次试算多个渠道，继续使用 `运营试算` 视图；买家模拟的手动指定更贴近真实下单，所以按单仓库、单客户渠道处理。
- 手动指定仍然要校验买家可见性、报价方案命中、客户渠道状态、客户渠道到系统渠道映射、系统渠道仓库绑定和上游能力。
- 手动指定允许展示需要客户上传面单的客户渠道；是否能继续试算由后端候选解析结果给出。

### 自动最优模式

字段：

- `限制仓库`：多选，非必填。

说明：

- 不选择仓库时，默认所有候选仓库都参与自动最优。
- 选择一个或多个仓库时，只在这些仓库范围内自动选最优客户渠道。
- 自动模式不展示渠道字段，客户渠道由系统自动选择。
- 自动模式应排除无法自动执行的客户渠道，例如必须客户上传面单的渠道。

## 买家模拟右侧看板

### 候选解析看板

右上区域展示系统从输入条件到候选结果的筛选过程。这个看板的价值是排障和解释，不只是展示最终费用。

建议展示以下解析节点：

| 节点 | 展示内容 |
| --- | --- |
| 商品仓库候选 | 每个 SKU 可发仓库、多个 SKU 的共同可发仓库、被限制仓库过滤掉的仓库。 |
| 仓库一致性 | 候选仓库的国家、币种是否一致。 |
| 报价方案 | 命中的 `BILLING` 报价方案、优先级、未命中原因。 |
| 客户渠道 | 方案内客户渠道、买家可见性、渠道状态、自动模式是否排除。 |
| 系统渠道 | 客户渠道映射到哪些系统渠道，系统渠道是否启用，是否绑定候选仓库。 |
| 上游能力 | 报价仓配对、履约仓配对、物流商渠道映射、外部试算能力是否满足。 |

展示方式：

- 用项目现有 Ant Design 组件实现，不复制截图样式。
- 每个节点显示数量和状态，例如 `可用 3 / 失败 2`。
- 失败原因保留在列表或展开明细里，便于看到具体卡在哪里。
- 空状态要撑开可视区域，避免右侧下半屏大面积空白。

### 试算结果看板

右下区域展示最终候选行和费用结果。

建议列：

| 列 | 说明 |
| --- | --- |
| 推荐 | 自动最优选中的行显示推荐标记；手动指定模式可为空。 |
| 仓库 | 实际发货仓库。 |
| 客户渠道 | 买家侧看到和选择的渠道。 |
| 买家费用 | 买家应付总金额。 |
| 系统渠道 | 系统内部实际执行渠道。 |
| 系统成本 | 系统内部预计成本。 |
| 差额 | 买家费用减系统成本；真实成本未接通时显示 `--`。 |
| 包裹尺寸 | 合并后的包裹尺寸，例如 `26 * 15 * 30 cm`。 |
| 实重 | 包裹实际重量。 |
| 体积重 | 根据尺寸计算的体积重。 |
| 计费重 | `max(实重, 体积重)`。 |
| 状态 | 成功、不可执行、外部试算失败、字段未确认等。 |

明确删除 `包裹数量` 列。当前口径为一个订单合并成一个包裹，默认包裹数为 1，不作为主要结果字段展示。

## 包裹尺寸和重量口径

买家模拟的商品包裹尺寸沿用现有费用试算 SKU 合包规则：

1. 每个 SKU 的三边先从小到大排序。
2. 包裹边 1 = 所有 SKU 最小边按数量相加。
3. 包裹边 2 = 所有 SKU 次长边最大值。
4. 包裹边 3 = 所有 SKU 最长边最大值。
5. 实重 = 所有 SKU 重量按数量相加。
6. 体积重 = 三边体积 / 当前系统体积重 divisor。
7. 计费重 = 实重和体积重取大。

当前没有商城确认尺寸字段，所以买家模拟第一阶段仍使用来源商品仓库测量尺寸。后续商城尺寸字段落地后，需要把 SKU 尺寸来源切换为商城确认尺寸。

## 自动最优业务链路

自动最优模式按以下顺序解析：

1. 输入买家、SKU 行、目的地、可选限制仓库。
2. 根据 SKU 所属 SPU 解析可发仓库。
3. 多 SKU 订单取共同可发仓库交集。
4. 如果填写了限制仓库，只保留限制仓库与共同可发仓库的交集。
5. 校验候选仓库国家和币种必须一致；不一致直接返回配置异常。
6. 匹配启用中的 `BILLING` 报价方案：
   - 买家范围命中。
   - 仓库范围命中。
   - 生效时间命中。
   - 状态启用。
7. 取报价方案里的客户渠道，并叠加客户渠道自身状态和买家可见规则。
8. 自动模式排除无法自动执行的客户渠道，例如必须客户上传面单的渠道。
9. 对每个客户渠道查找映射系统渠道。
10. 校验系统渠道启用、系统渠道绑定候选仓库。
11. 外部试算前预检查报价仓配对、履约仓配对、物流商渠道映射。
12. 对剩余候选调用外部试算；单个候选失败只淘汰该候选，全部失败才让本次自动最优失败。
13. 买家侧按买家应付总额最低选最优客户渠道。
14. 客户渠道确定后，在映射系统渠道里按系统成本最低选择最终系统渠道。

如果当前外部费用试算字段仍未确认，系统必须 fail-closed，展示明确失败原因，不允许前端或后端伪造金额。

## 手动指定业务链路

手动指定模式按以下顺序解析：

1. 输入买家、SKU 行、目的地、指定仓库、指定客户渠道。
2. 校验指定仓库必须在所有 SKU 的共同可发仓库内。
3. 匹配启用中的 `BILLING` 报价方案。
4. 校验指定客户渠道在命中的报价方案内。
5. 校验指定客户渠道对当前买家可见。
6. 查找客户渠道映射的系统渠道。
7. 校验系统渠道启用、绑定指定仓库。
8. 校验上游配对和物流商渠道映射。
9. 调用外部试算或返回明确失败原因。

手动指定不是绕过规则，只是把仓库和客户渠道从“系统选择”改成“用户指定”。

## 后端改造计划

### 接口形态

建议继续归属现有管理端费用试算接口前缀：

- `GET /finance/admin/fee-estimate/options`
- `GET /finance/admin/fee-estimate/skus/list`
- `POST /finance/admin/fee-estimate/calculate`

在请求体里明确增加视图和买家模拟字段，而不是新增无关菜单：

- `estimateView`：`OPERATIONS` / `BUYER_SIMULATION`
- `selectionMode`：`MANUAL` / `AUTO_BEST`
- `buyerId`
- `originWarehouseCode`
- `warehouseCodes`
- `customerChannelCode`
- `packageInputMode`
- `packageLines`
- `destinationCountryCode`
- `destinationState`
- `destinationCity`
- `destinationPostalCode`
- `destinationAddress1`
- `destinationAddress2`

说明：

- `运营试算` 可继续使用现有字段。
- `买家模拟 + 手动指定` 使用 `originWarehouseCode + customerChannelCode`。
- `买家模拟 + 自动最优` 使用 `warehouseCodes` 作为可选限制仓库；空数组表示全部候选仓库。
- 如果当前代码已有 `selectionMode` 和自动最优雏形，下一步重点是把页面视图语义拆清楚，并把请求字段从单仓库扩展为自动模式下的多仓库限制。

### 服务分层

继续遵守当前复用边界：

- `finance` 负责编排试算。
- 商品尺寸通过 `FinanceFeeEstimateSkuLookupService` 读取。
- 买家、仓库、客户渠道、系统渠道通过 lookup port 或已有 service/facade 读取。
- `finance` 不直接依赖 product、warehouse、logistics、integration 的 Mapper。
- 领星或物流商外部试算通过 integration/logistics 适配层接入，不在 finance Controller 里拼 HTTP。

### 响应结构

响应建议包含：

- `requestNo`
- `estimateView`
- `selectionMode`
- `buyerSnapshot`
- `packageSummary`
- `skuSnapshots`
- `resolveSummary`
- `routeCandidates`
- `results`

`resolveSummary` 用于右上候选解析看板。

`results` 用于右下试算结果表。

   - Tabs 存在。
   - 买家模拟字段存在。
   - 自动最优不展示渠道字段。
   - 手动指定展示仓库和客户渠道字段。

## 当前风险

1. 运行库里部分报价方案仓库明细可能不完整，严格规则下自动最优可能在报价方案匹配阶段失败。
2. 领星费用试算接口字段仍未完全确认，真实金额可能继续 fail-closed。
3. 部分仓库可能只有履约配对，没有报价配对，需要在候选解析里清楚展示失败原因。
4. 自动最优如果候选配置过宽，需要先通过 SQL 和内存规则压缩候选，不能直接按 `仓库 * 客户渠道 * 系统渠道` 暴力调用外部接口。

## 成功标准

1. 费用试算页面有 `运营试算` 和 `买家模拟` 两个视图。
2. `买家模拟` 左侧字段符合确认口径。
3. `手动指定` 展示仓库和客户渠道字段。
4. `自动最优` 只展示可选限制仓库，不展示渠道字段。
5. 右上候选解析能解释每一步通过和失败原因。
6. 右下结果表展示仓库、客户渠道、买家费用、系统渠道、系统成本、差额、包裹尺寸、实重、体积重、计费重和状态。
7. 不展示包裹数量列。
8. 不新增表，不写历史，不伪造费用金额。
